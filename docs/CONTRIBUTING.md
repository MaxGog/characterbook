# Contributing to CharacterBook

Спасибо, что хотите помочь развитию CharacterBook! Этот документ содержит правила и рекомендации, которые помогут нам совместно работать над проектом.

## Прежде чем начать

- Поищите [открытые issues](https://github.com/maxgog/characterbook/issues), возможно, ваша идея или ошибка уже обсуждается.
- Если вы хотите предложить новую функцию или крупное изменение, сначала создайте issue для обсуждения, чтобы не тратить время впустую.
- Для мелких правок (опечатки, комментарии) можно сразу отправлять Pull Request.

## Настройка окружения

### Требования

- Flutter 3.13+ (стабильная версия)
- Dart 3.7.2+
- Любая IDE с поддержкой Flutter (рекомендуем VS Code или Android Studio)

### Установка

1. Форкните репозиторий.
2. Склонируйте свой форк:
   ```bash
   git clone https://github.com/<your-username>/characterbook.git
   cd characterbook
   ```
3. Установите зависимости:
   ```bash
   flutter pub get
   ```
4. Сгенерируйте код (сериализация, локализация, ассеты):
   ```bash
   dart run build_runner build --delete-conflicting-outputs
   ```
5. Проверьте, что всё работает:
   ```bash
   flutter run
   ```

## Процесс внесения изменений

### Ветки

- `main` — стабильная ветка, готовая к релизу.
- Создавайте ветки от `main`:
  - `feature/название-фичи` — для новых возможностей.
  - `fix/название-бага` — для исправлений.
  - `docs/описание` — для изменений в документации.

Пример: `feature/pdf-export-preview` или `fix/character-delete-crash`.

### Коммиты

Следуем соглашению [Conventional Commits](https://www.conventionalcommits.org/):

- `feat:` — новая функциональность
- `fix:` — исправление ошибки
- `docs:` — документация
- `refactor:` — рефакторинг без изменения логики
- `test:` — тесты
- `chore:` — рутинные задачи (обновление зависимостей и т.п.)

Пишите сообщения на английском языке, в настоящем времени: «add character export» вместо «added character export».

## Стиль кода

Мы придерживаемся [официального стиля Dart](https://dart.dev/effective-dart) и используем анализатор Flutter.

Обязательно выполните перед коммитом:

```bash
flutter analyze
dart format lib test
```

При CI-сборке эти проверки запускаются автоматически, и PR без их прохождения не будет принят.

## Архитектура проекта

CharacterBook следует принципам чистой архитектуры. Перед тем как писать код, ознакомьтесь с [ARCHITECTURE.md](./docs/ARCHITECTURE.md). Краткая структура:

- `lib/core/` — базовые сущности
- `lib/data/` — модели, репозитории, сервисы, перечисления
- `lib/handlers/` — платформозависимые обработчики файлов
- `lib/interfaces/` — контракты
- `lib/providers/` — управление состоянием (Provider, Bloc)
- `lib/services/` — бизнес-логика и внешние сервисы
- `lib/ui/` — экраны, виджеты, контроллеры UI

Старайтесь размещать новые файлы в правильных директориях и следовать существующим паттернам (например, одна модель — один файл, репозиторий над Hive-боксом).

## Тестирование

Мы ожидаем, что новая функциональность будет покрыта тестами.

### Типы тестов

1. **Unit-тесты** (`test/unit/`) — изолированная проверка логики репозиториев, сервисов, провайдеров.
2. **Widget-тесты** (`test/widget/`) — проверка UI-компонентов с замоканными зависимостями.
3. **Интеграционные тесты** (`integration_test/`) — сквозные сценарии в реальном окружении.

### Запуск

```bash
# Все тесты
flutter test

# С покрытием
flutter test --coverage

# Интеграционные тесты
flutter test integration_test
```

В идеале пишите тесты одновременно с кодом. Если вы исправляете баг, добавьте тест, который воспроизводит проблему.

## Pull Request

1. Убедитесь, что ваш код проходит `flutter analyze`, отформатирован и все тесты зелёные.
2. Обновите документацию, если изменили API или архитектуру.
3. Создайте Pull Request в ветку `main` вашего форка.
4. Опишите, что сделано, и сошлитесь на связанный issue (например, «Closes #42»).
5. Дождитесь проверки CI и ревью от мейнтейнеров. Будьте готовы внести правки по замечаниям.

Шаблон описания PR:

```
## Описание
Кратко о том, что было сделано.

## Связанные задачи
Closes #номер_issue

## Тип изменений
- [ ] Исправление бага
- [ ] Новая функция
- [ ] Изменение существующей функциональности
- [ ] Документация

## Как тестировалось
Опишите, как проверяли работоспособность.

## Чеклист
- [ ] Код прошёл `flutter analyze`
- [ ] Добавлены/обновлены тесты
- [ ] Документация обновлена
```

## Дополнительные рекомендации

- **Локализация**: все пользовательские строки должны использовать `AppLocalizations`. Добавляйте ключи в `lib/l10n/intl_en.arb` и `intl_ru.arb`.
- **Адаптеры Hive**: при создании новой модели не забудьте зарегистрировать адаптер и запустить генерацию кода (`build_runner`).
- **Изображения и ассеты**: кладите в `assets/` и используйте `flutter_gen` для типобезопасного доступа.
- **Зависимости**: прежде чем добавить новый пакет, обсудите это в issue. Мы стараемся минимизировать внешние зависимости.
